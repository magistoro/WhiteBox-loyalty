# WhiteBox - рабочий кабинет компании

Актуально на 24 мая 2026 года. Документ описывает первый рабочий контур кабинета партнера: команду, кассу, подписки, погашения и финансовый прогноз.

## Граница ролей

Роли администрации WhiteBox и роли сотрудников партнера не пересекаются.

- `User.role = COMPANY` открывает портал компании на уровне платформы.
- `CompanyMember` определяет доступ сотрудника внутри конкретной компании.
- `OWNER` управляет командой, тарифами, погашениями и финансами компании.
- `MANAGER` управляет операциями и кассирами, но не может назначать или отключать других руководителей.
- `CASHIER` обслуживает клиента: находит его по QR, начисляет баллы и погашает доступные выдачи по подписке.
- Владелец компании не отключается из интерфейса команды; такие изменения потребуют отдельного защищенного процесса.

Один пользователь в будущем сможет быть сотрудником нескольких компаний без смешивания прав и балансов: членство всегда связано с `companyId`.

## Интерфейс компании

| Страница | Назначение | Доступ |
|---|---|---|
| `/company` | Дашборд с клиентами, подписчиками, прогнозом и последними покупками | Все сотрудники |
| `/company/clients` | Касса: QR/поиск клиента, начисление баллов, погашение подписки | Все сотрудники |
| `/company/subscriptions` | Тарифы и правила выдачи услуг по подписке | Просмотр всем, изменение `OWNER`/`MANAGER` |
| `/company/team` | Приглашение сотрудников и управление статусом кассиров | `OWNER`/`MANAGER` с ограничениями ролей |
| `/company/payments` | Прогноз подписочной выручки и заявки на выплату | `OWNER`/`MANAGER` |
| `/company/compliance` | Профиль компании и статус верификации | Все сотрудники |

## Клиент и начисление баллов

Касса поддерживает два сценария:

1. Ручное начисление фиксированного количества баллов, например компенсации или бонуса.
2. Начисление от покупки: сотрудник вводит сумму, API определяет уровень клиента в этой компании и начисляет кэшбэк по `CompanyLevelRule`.

Пример: у клиента уровень `Bronze` с кэшбэком `1%`; при покупке на `1000` начисляется `10` баллов. При уровне `Silver` с кэшбэком `5%` та же покупка дает `50` баллов.

Сумма покупки и вычисленный процент фиксируются в `CompanyPurchase`, а движение баллов - в `LoyaltyTransaction` и балансе `UserCompany`. Начисление баллов, покупка в кассе и оформление подписки создают связь `UserCompany`, поэтому клиент сразу появляется в поиске своей компании. Расчет уровня и запись покупки выполняются в транзакции уровня `Serializable`, чтобы два одновременных чека не рассчитали неверный уровень.

Покупка подписки учитывается в прогрессе уровня и начислении баллов в соответствии с настройкой `subscriptionSpendPolicy`: исключить подписки, учитывать только траты или учитывать траты с кэшбэком.

### Видимость персональных данных

- Обычный поиск по имени или email возвращает только клиентов, у которых уже есть связь, покупка или активная подписка этой компании.
- QR может открыть нового клиента для обслуживания, но email нового клиента до появления связи не раскрывается.

## Подписки и погашения

Тариф остается привязан к одной компании. Для каждой услуги тарифа создается `SubscriptionEntitlement`:

- `title` и `description` описывают выдаваемую услугу;
- `allowance` задает количество выдач;
- `windowValue` и `windowUnit` задают период лимита: `DAY`, `WEEK`, `MONTH` или `TERM`.
- `UNLIMITED` отключает ограничение по количеству использований, сохраняя каждое посещение в истории; это подходит, например, для доступа в фитнес-клуб.

Примеры:

- `1` кофе каждые `1 DAY` - одна выдача в сутки;
- `10` пачек чая каждые `1 MONTH` - до десяти выдач за расчетный месяц;
- доступ в фитнес-клуб с `UNLIMITED` - входы фиксируются для статистики без блокировки повторного посещения;
- `1` консультация за `TERM` - одна выдача на весь срок подписки.

Погашение записывается в `SubscriptionRedemption`. API проверяет активную подписку, активное правило и оставшийся лимит; проверка и запись защищены `Serializable`-транзакцией, поэтому два кассира не должны одновременно выдать услугу сверх лимита.

Клиентское приложение показывает настоящее описание тарифа и список `SubscriptionEntitlement`, а история клиента различает покупку подписки, пополнение баллов и списание баллов.

## Финансы

Дашборд и финансовая страница показывают:

- активных подписчиков;
- сумму номиналов активных тарифов;
- текущий признанный доход `recognizedSubscriptionRevenue`: долю стоимости за уже завершенные дни оформленных подписок; уже заработанный доход сохраняется после окончания срока;
- потенциальный оставшийся доход `potentialSubscriptionRevenue` и темп ежедневного признания `dailySubscriptionRevenue`;
- доступно к выводу `availableForPayout`: признанный доход минус уже оплаченные выплаты и заявки, зарезервированные в статусах `PENDING_APPROVAL`/`APPROVED`;
- покупки и начисленные баллы;
- заявки на выплату, ожидающие рассмотрения.

Пример: подписка `1000` на `10` дней и подписка `365000` на `365` дней после одного завершенного дня дают `1100` текущего дохода и `364900` потенциального. Запрос на выплату создает резерв `FinanceOperation` со статусом `PENDING_APPROVAL`; API компании не позволяет зарезервировать больше доступной суммы, а API администратора повторно проверяет покрытие при одобрении и оплате. Успешный запрос компании также отправляет персональное уведомление привязанным активным администраторам через Telegram queue; временный сбой Telegram не отменяет созданную финансовую заявку.

Решение по заявке на верификацию является конечным: после `APPROVED` или `REJECTED` статус нельзя изменить повторным действием администратора. Это ограничение проверяется сервером в сериализуемой транзакции, а не только скрытием кнопок в интерфейсе.

Сейчас финансовым источником является факт активации подписки клиента. Это прозрачный расчётный баланс для разработки, но не банковское подтверждение поступления денег. Перед реальными выплатами нужен отдельный payment ledger с успешными списаниями платёжного провайдера, комиссиями, возвратами и налоговыми удержаниями; расчёт вывода следует перевести на него.

## Данные и резервные копии

Новые операционные модели:

- `CompanyMember` - сотрудники и локальные роли компании;
- `CompanyPurchase` - покупки, от которых начислен кэшбэк;
- `SubscriptionEntitlement` - правила доступных выдач;
- `SubscriptionRedemption` - факты погашения услуги.

Snapshot schema version `2` включает эти модели и `FinanceOperation`. Восстановление удаляет и создает записи в порядке зависимостей и восстанавливает PostgreSQL sequences для новых таблиц.

Для применения схемы после получения изменений:

```bash
npm run db:generate
npm run db:migrate
```

## Проверки

API unit tests закрепляют:

- начисление фиксированных баллов без ложной покупки;
- начисление кэшбэка по достигнутому уровню;
- `Serializable`-защиту начислений и погашений;
- запрет повторной выдачи сверх периодического лимита;
- запрет кассиру менять тарифы и менеджеру отключать другого менеджера;
- ограничение поиска клиентов связями конкретной компании;
- появление клиента после покупки без начисленного кэшбэка;
- сохранение компанией политики уровней и баллов от подписок;
- ежедневное признание подписочной выручки и оставшегося потенциала;
- сохранение признанного дохода завершённой подписки и запрет вывода сверх заработанного свободного остатка;
- попадание новых таблиц рабочего кабинета в backup snapshot.

Полная локальная проверка:

```bash
npm run ci:verify
```

## Безопасный пересев тарифов компании

Основной `npm run db:seed` пересоздает демонстрационные аккаунты и не должен использоваться для сброса тарифов на рабочей базе. Для миграции тестовых подписок на систему правил выдачи предусмотрена ограниченная команда:

```bash
npm run db:seed:company-subscriptions -- --dry-run
npm run db:seed:company-subscriptions -- --apply
```

Первый вызов показывает масштаб операции без изменения БД. Вызов с `--apply` удаляет только тарифы, принадлежащие компаниям, связанные с ними клиентские подписки/погашения и парные предложения, затем создает два тарифа с `SubscriptionEntitlement` для каждой существующей компании. Пользователи, компании, балансы и глобальные тарифы не удаляются.

## Важное ограничение перед продакшеном

Текущий QR клиента содержит стабильный payload вида `whitebox:user:<uuid>`. Он удобен для текущего UX, но скриншот QR можно повторно показать кассиру. Перед реальными выдачами услуг нужно согласовать и внедрить короткоживущий подписанный QR (например, сроком 60-90 секунд) либо иной сценарий подтверждения клиента.
